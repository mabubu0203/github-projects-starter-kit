## コアメッセージ

### 中核メッセージ

GitHub Projects Ops Kit は、**GitHub Projects の運用立ち上げと継続運用を支援するプロダクト**です。

Project の作成・Field / Status / View の設定から、Issue / PR の紐付け、滞留検知、レポート生成までを
GitHub Actions の Workflow 実行だけで完結させます。

### サブメッセージ

| 対象 | メッセージ |
|------|-----------|
| 全般 | Fork して Workflow を実行するだけ。JSON 定義ファイルで構成を管理し、何度でも同じ環境を再現できます。 |
| 構築面 | Project の新規作成から Field・Status・View のセットアップまで、手作業なら 30 分以上かかる作業が約 1 分で完了します。 |
| 運用面 | 滞留タスクの検知、サマリー・工数・ベロシティレポートの生成で、プロジェクトの健康状態を定量的に把握できます。 |
| 標準化 | JSON 定義ファイルをテンプレートとして共有すれば、チーム全体で統一された Project 構成を維持できます。 |

## 想定顧客セグメント

### セグメント A: 個人開発者 / ソロメンテナー

**ペルソナ:** 個人で OSS や趣味プロジェクトを運営している開発者。GitHub は日常的に使うが、Project 機能は手付かずか、途中で管理が崩れてしまう。

| 項目 | 内容 |
|------|------|
| **課題** | GitHub Projects の初期設定が面倒 / 運用の型を自分で作るのが大変 / 途中で管理が崩れやすい |
| **刺さる価値** | すぐ始められる / 再現性のある管理ができる / 面倒な初期設定を省ける |
| **代表ユースケース** | OSS v1.0 リリース準備で Issue を整理し、Board で進捗を可視化する / 複数の個人リポジトリを 1 つの Project で一元管理する |
| **訴求キーワード** | 「Fork して実行するだけ」「約 1 分でセットアップ完了」「無料で使える」 |

### セグメント B: OSS メンテナー / 小規模チーム

**ペルソナ:** 2〜5 名程度のチームで開発を行う OSS メンテナーや小規模チーム。コントリビュータが増えてきて運用ルールの統一が課題になり始めている。

| 項目 | 内容 |
|------|------|
| **課題** | コントリビュータが増えると運用ルールが曖昧になる / Project の見える化を整えたい / Issue / PR 管理を標準化したい |
| **刺さる価値** | 運用ルールの共通化 / Project の整備・可視化 / 追加・集計・分析の自動化 |
| **代表ユースケース** | JSON 定義ファイルで Field・Status・Label を標準化し、誰が Project を作っても同じ構成にする / Issue/PR を一括で Project に紐付け、Board で全体像を把握する |
| **訴求キーワード** | 「構成をコードで管理」「Label も一括設定」「チーム全員が同じ画面で作業」 |

### セグメント C: Organization 管理者 / PM / EM / TL

**ペルソナ:** Organization 配下の複数リポジトリ・複数プロジェクトを横断的に管理する立場。定例報告やメンバーの稼働状況の可視化に課題を感じている。

| 項目 | 内容 |
|------|------|
| **課題** | GitHub Projects を導入しても継続運用が定着しない / レポート作成が手作業 / チーム横断の状況把握がしづらい |
| **刺さる価値** | 標準化された運用基盤 / 分析・エクスポートの効率化 / マネジメントに使えるデータ出力 |
| **代表ユースケース** | 新チーム立ち上げ時に Workflow 実行だけで Project 環境を整備する / 週次で分析 Workflow を実行し、サマリー・ベロシティレポートを定例報告に活用する |
| **訴求キーワード** | 「立ち上げ 10 分」「レポート自動生成」「外部ツール不要でコスト削減」 |

## 流入元別 訴求方針

### GitHub 流入（README / リポジトリページ）

**想定読者:** 開発者・技術者が中心。GitHub 上でリポジトリを見つけて訪問する。

| 優先度 | 伝える内容 |
|--------|-----------|
| 高 | 一言説明（何ができるか）→ 機能一覧 → クイックスタート |
| 中 | Before/After の比較表 → Workflow の実行イメージ |
| 低 | 詳細なユースケース・ブランドメッセージ |

**方針:**
- 「何ができるか」を最初の 3 行で伝える
- 技術的な正確性を重視し、具体的な機能名・Workflow 名を記載する
- 「Fork → PAT 設定 → Workflow 実行」の 3 ステップを明示する
- Star / Fork を促す導線を意識する

### Docs 流入（GitHub Pages）

**想定読者:** すでに興味を持っている利用者・検討者。検索や README のリンクから訪問する。

| 優先度 | 伝える内容 |
|--------|-----------|
| 高 | ユースケース別ガイド → やりたいこと別ガイド → クイックスタート |
| 中 | 課題解決の具体例（Before/After）→ シナリオ別の活用方法 |
| 低 | 開発者向けリファレンス |

**方針:**
- 「自分の状況に合った使い方」を見つけやすい構成にする
- セグメント別のユースケースを前面に出す
- ステップバイステップの導入手順を丁寧に示す
- 困ったときの FAQ・トラブルシューティングへの導線を確保する

以下は README / Docs / SNS 等で流用できる文章素材です。

### 短い紹介文（1〜2 文）

> GitHub Projects の初期構築から運用分析までを GitHub Actions で半自動化する運用キットです。Fork して Workflow を実行するだけで、Project の作成・分析・レポート生成を一気通貫で行えます。

### やや詳しい紹介文（3〜5 文）

> GitHub Projects Ops Kit は、GitHub Projects の運用立ち上げと継続運用を支援するツールキットです。Project の作成、Field・Status・View の一括セットアップ、Issue/PR の紐付け、滞留検知、各種レポート生成までを GitHub Actions の Workflow で実行できます。JSON 定義ファイルで構成を管理するため、何度でも同じ環境を再現でき、チーム内での標準化にも活用できます。個人開発者から Organization 管理者まで、GitHub Projects を本格的に運用したいすべての方に向けたキットです。

### 特徴を伝える箇条書き

- **即時セットアップ:** Workflow を 1 回実行するだけで、Project の作成から Field・Status・View の設定まで完了
- **構成のコード管理:** JSON 定義ファイルでプロジェクト構成を管理し、再現性と標準化を実現
- **運用分析の自動化:** 滞留検知・サマリー・工数・ベロシティレポートを自動生成
- **GitHub 完結:** 外部ツール不要。GitHub Actions と GitHub CLI だけで動作
- **個人にもチームにも:** 個人開発者の手軽な管理から、Organization の本格運用まで対応
